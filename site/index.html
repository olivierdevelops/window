📸 Photo Booth

Live webcam preview, a spoken countdown, and a snapshot to PNG.

cameraspeechinput

entry_path: ./static/index.html

shoot.onclick = async () => {
  await NATIVE.speech.say("Three. Two. One.");
  img.src = await NATIVE.camera.snapshot();
};

the .window app language

Write a whole app in one tiny file

window has its own declarative app language — a .window file. Run it with window app.window: it transpiles to the UI and the window.yaml, then opens. (The transpiler under the hood is the Capy engine — you never see it.) Backend calls are served in-process by Go handlers built into the binary — no Python, no Node, no socket, no sidecar.

one source · one binary
🔢 counter.window — 11 lines

State, events and reactive bindings — projected into a full app.

.windowno backend
counter.window
app "Counter" 320 240
    heading "Counter"
    text count
    button "−" id dec
    button "+" id inc
    state count = 0
    on "#inc" click do up
    on "#dec" click do down
    action up   set count = state.count + 1
    action down set count = state.count - 1
end
generated app.js (excerpt)
const state = { count: 0 };
const ACTIONS = {
  up:   () => setState({ count: state.count + 1 }),
  down: () => setState({ count: state.count - 1 }),
};
document.querySelector("#inc")
  .addEventListener("click", () => ACTIONS.up());
render();
Run it

backends

A backend in any language

Set run_backend_script and window launches that process, handing it a socket path. The two sides exchange newline-delimited JSON — which is why the backend can be Python, Node, Go, Rust, or anything else.

main.py
from client import App        # tiny helper shipped with window
app = App()

@app.handle("add")
async def add(req, rw):
    await rw.send({ "value": req["value"] + 1 })

app.run()
main.js
const net = require("net");
const sock = net.connect(process.env.WINDOW_SOCK_PATH);
sock.on("data", buf => {
  const m = JSON.parse(buf);
  sock.write(JSON.stringify({ id: m.id, function: m.function,
    data: { value: m.data.value + 1 }, done: true }) + "\n");
});
main.go
conn, _ := net.Dial("unix", os.Getenv("WINDOW_SOCK_PATH"))
dec, enc := json.NewDecoder(conn), json.NewEncoder(conn)
for {
    var m Msg; dec.Decode(&m)
    m.Data["value"] = m.Data["value"].(float64) + 1
    enc.Encode(m)
}
main.rs
let mut sock = UnixStream::connect(env::var("WINDOW_SOCK_PATH")?)?;
for line in BufReader::new(sock.try_clone()?).lines() {
    let mut m: Value = serde_json::from_str(&line?)?;
    m["data"]["value"] = json!(m["data"]["value"].as_i64().unwrap() + 1);
    writeln!(sock, "{}", m)?;
}
Frontend: BACKEND.call("add", {value:41}, …) →
{ value: 42 }   // same result, whichever language you chose

how it works

A config in. A native window out.

🏗 Architecture

window follows a strict layered design (VHCO): data flows one direction and every dependency is injected, so each piece is testable in isolation.

package layout
domain/        what the app is        (AppConfig, Message, RunMode)
features/      what it can do         (structs of capability functions)
usecases/      how it does things     (run modes, init-window)
infra/         external impls         (webview, sockets, http, wasm, native/*)
orchestrator/  the wiring             (builds & injects everything)
appio/         CLI + config loading

Full write-up: docs/architecture.md

🔌 Backend protocol

Newline-delimited JSON over a Unix socket. Each message has an id; the final reply sets done:true; a backend pushes events using an id prefixed with server:.

messages on the socket
→  {"id":"a1","function":"add","data":{"value":5},"done":true}   // frontend asks
←  {"id":"a1","function":"add","data":{"value":6},"done":true}   // backend replies
←  {"id":"server:x","function":"tick","data":{"t":"12:30"}}      // backend pushes

Spec: docs/backend-protocol.md. Test with go test ./....

comparison

window vs Electron, Tauri & Wails

All four put a web UI in a desktop window. They differ most in what language the backend is and how much tooling you take on.

 windowElectronTauriWails
Backend languageany (socket / WASM / none)Node.jsRustGo
Bundles a browsernoyes (Chromium)nono
App sizesmall~100–200 MBsmallsmall
Build toolchainnone — a YAML filebundler + NodeRust + cargoGo + CLI
Device APIsNATIVE.* built-inNode + nativeRust commandsGo bindings
Wrap existing dev serveryes (proxy)manualmanualmanual

Pick window when…

  • your backend already exists in Python, Go, Rust, or anything non-Node;
  • you want to ship or generate apps as plain files with no build step;
  • you want camera/mic/speech/screen/input as one-liners.
Honest trade-off: window uses the system webview (like Tauri/Wails), so rendering can differ slightly per OS, and its ecosystem is younger than Electron's — in exchange you get tiny apps and total backend-language freedom.
