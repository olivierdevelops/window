how it works

A config in. A native window out.

There is no SDK to learn and no build pipeline to wire up. You write a UI in plain HTML/CSS/JS, describe the app in a window.yaml file, and run the window binary against it. The binary serves your files to the operating system's built-in webview and — optionally — launches a backend process it talks to over a local socket.

why it exists

What window brings to the table

Most desktop-web frameworks marry you to one backend language — Node for Electron, Rust for Tauri, Go for Wails. window takes a different stance: the backend is a separate process that speaks a tiny JSON protocol over a local socket, so it can be written in any language — or compiled to WASM, or skipped entirely. Three ideas define the project:

  • Language-agnostic backends. Python today, Go or Rust tomorrow — no FFI, no rewrite. If it can open a socket and parse JSON, it can be a window backend.
  • Declarative, no build step. An app is a folder with a window.yaml. No bundler, no native toolchain, no node_modules. That makes apps trivial to share, script, and even generate with AI.
  • Batteries-included device APIs. Camera, mic, speech, screen and input are one-liners on window.NATIVE, alongside file system, shell, dialogs and canvas.
Honest trade-off: window uses the operating system's webview (like Tauri and Wails), so rendering can differ slightly across platforms, and its ecosystem is younger than Electron's. In exchange you get tiny apps, no bundled browser, and total freedom in backend language. Jump to the full comparison →

device capabilities

The window.NATIVE object

Every native capability hangs off one global, window.NATIVE, and is injected before your page's JavaScript runs. You opt in per app by listing features in window.yaml — nothing is exposed unless you ask for it.

window.yaml
native_features:
  - camera   # window.NATIVE.camera.*
  - speech   # window.NATIVE.speech.*
  - input    # window.NATIVE.input.*
cameraphotos · video →
micrecord · meter →
speechTTS · STT →
screencapture · record →
inputkeys · mouse →

📷 camera

Take a still photo, mirror a live preview into a <video>, or record a clip. The camera is opened, used, and released automatically — no track teardown to forget, no lingering camera light.

app.js
// Snapshot: opens the camera, grabs one frame, releases it
const photo = await NATIVE.camera.snapshot({ width: 640, type: "image/jpeg" });
document.querySelector("img").src = photo;
Returns
"data:image/jpeg;base64,/9j/4AAQSkZJRgABAQAAAQABAAD…"  // a 640px-wide JPEG

Other calls: NATIVE.camera.stream({ into:'#cam' }) for a live preview, NATIVE.camera.record({ ms:5000 }) for a clip, and NATIVE.camera.list() to enumerate devices.

🎙 mic

Record audio to a playable clip, or read a live input level for a VU meter — handy for a "is it hearing me?" indicator. Recording returns a small handle with a stop() that resolves to the finished audio.

app.js
const rec = await NATIVE.mic.record();
stopButton.onclick = async () => {
  player.src = await rec.stop();        // resolves when recording stops
};

// Live level meter (0.0 – 1.0 every animation frame)
await NATIVE.mic.meter(level => bar.style.width = (level * 100) + "%");
rec.stop() returns
"data:audio/webm;base64,GkXfo59ChoEBQveBAULygQRC…"   // a webm audio clip

🔊 speech

Two directions: say() speaks text aloud and resolves when finished, so you can await a line before continuing. listen() streams the microphone to text, emitting interim results as the user speaks and marking each chunk isFinal when committed.

app.js
await NATIVE.speech.say("Your build is complete.", { rate: 1.1 });

const session = NATIVE.speech.listen(({ text, isFinal }) => {
  caption.textContent = text;
  if (isFinal) transcript.value += text + " ";
});
listen() callback fires
{ text: "your build is",        isFinal: false }
{ text: "your build is complete", isFinal: true }
Heads-up: say() is broadly supported; listen() (speech-to-text) depends on the platform webview and throws where unavailable — feature-check before building a voice-only flow.

🖥 screen

Screenshot or record the screen. The OS shows a one-time picker so the user chooses exactly what to share — by design it can't silently grab the whole desktop.

app.js
const shot = await NATIVE.screen.snapshot();          // PNG of the chosen surface

const rec = await NATIVE.screen.record({ audio: true });
stopBtn.onclick = async () => download.href = await rec.stop();
snapshot() returns
"data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA…"   // after the user picks a screen

⌨️ input

Bind hotkeys, watch raw key events, or track the mouse — everything an app needs for shortcuts, canvas tools and games. Hotkeys handle preventDefault for you.

app.js
NATIVE.input.hotkey("ctrl+s", () => saveDocument());   // "cmd+s" alias on macOS
NATIVE.input.onKey(e => { if (e.key === "Escape") closeModal(); });
NATIVE.input.onMouse(e => trail.push([e.x, e.y]), { events: ["mousemove"] });
onKey() callback fires
{ key: "s", code: "KeyS", type: "keydown", ctrl: true, shift: false, alt: false, meta: false }

🧰 fs · os · dialogs · canvas

The original native features ship alongside the new device APIs. These are backed by Go, so they reach the real machine — files, the shell, native OS dialogs, and a 2D canvas.

app.js
await NATIVE.fs.readDir("/Users/me/projects");
await NATIVE.os.exec("git", ["status", "--short"]);
const [path] = await NATIVE.dialogs.openFile({ title: "Pick a file" });
fs.readDir() returns

backends

A backend in any language

When you set run_backend_script, window launches that process and hands it a socket path via the WINDOW_SOCK_PATH environment variable. The two sides exchange newline-delimited JSON. Your frontend calls BACKEND.call(name, params, cb); your backend answers. That's the whole contract — which is why the backend can be Python, Node, Go, Rust, or anything else.

1 · The backend — pick your language

main.py
from client import App        # tiny helper shipped with window
app = App()

@app.handle("add")
async def add(req, rw):
    await rw.send({ "value": req["value"] + 1 })

app.run()
main.js
const net = require("net");
const sock = net.connect(process.env.WINDOW_SOCK_PATH);

sock.on("data", buf => {
  const m = JSON.parse(buf);
  const reply = { id: m.id, function: m.function,
                  data: { value: m.data.value + 1 }, done: true };
  sock.write(JSON.stringify(reply) + "\n");
});
main.go
conn, _ := net.Dial("unix", os.Getenv("WINDOW_SOCK_PATH"))
dec := json.NewDecoder(conn)
enc := json.NewEncoder(conn)
for {
    var m Msg
    dec.Decode(&m)
    m.Data["value"] = m.Data["value"].(float64) + 1
    enc.Encode(m)   // newline-delimited JSON
}
main.rs
let path = std::env::var("WINDOW_SOCK_PATH")?;
let mut sock = UnixStream::connect(path)?;
let reader = BufReader::new(sock.try_clone()?);

for line in reader.lines() {
    let mut m: Value = serde_json::from_str(&line?)?;
    m["data"]["value"] = json!(m["data"]["value"].as_i64().unwrap() + 1);
    writeln!(sock, "{}", m)?;        // reply
}

Same wire format every time: read newline-delimited JSON, write it back. No SDK, no FFI.

2 · The frontend call

app.js
BACKEND.call("add", { value: 41 }, ({ data }) => {
  console.log(data.value);
});
Console
42

3 · The config that wires them

window.yaml
title: "Counter"
entry_path: ./static/index.html
run_backend_script: python3 main.py

The backend can also push events to the UI at any time (a clock, a progress bar, a chat message) — see the backend protocol below.

run modes

Bring any backend — or none

The mode field decides how the app is served and what drives it. The default (server mode) covers most apps; the rest let you wrap existing servers, ship a dependency-free WASM backend, or let the backend drive the windows.

server

HTTP + optional subprocess backend (default)

wasm

A WASM module is the backend — zero deps

controlled

Backend creates & drives the windows

url

Point straight at a remote URL

proxy

Wrap an existing dev server (Vite, Next…)

browser

Inject an overlay into any web page

⚙️ WASM backend

For a fully self-contained app with no subprocess and no runtime to install, compile your backend to WebAssembly and point wasm_backend at it. window loads it with wazero (pure Go, no CGo) and routes every BACKEND.call() through it. The frontend code is identical to the socket version — only the transport changes.

window.yaml
mode: wasm
wasm_backend: ./app.wasm
entry_path: ./static/index.html

Details & the module contract: docs/wasm-backend.md

Walkthrough · Photo Booth (no backend)

A complete camera app in two files. Enable the features, drop in a video and a button, wire three calls. There is no backend process at all.

window.yaml
title: "Photo Booth"
entry_path: ./static/index.html
native_features: [camera, speech, input]
static/index.html
<video id="cam" autoplay muted></video>
<img id="shot">
<button id="shoot">Take photo</button>
<script>
  NATIVE.camera.stream({ into: "#cam" });            // live preview
  shoot.onclick = async () => {
    await NATIVE.speech.say("Three. Two. One.");      // spoken countdown
    shot.src = await NATIVE.camera.snapshot();        // capture
  };
  NATIVE.input.hotkey("space", () => shoot.click());  // shortcut
</script>
Result
A native window showing the live webcam. Press Space or click "Take photo":
the app speaks a countdown, then freezes the captured photo in place.

for developers

How the binary is built

window follows a strict layered architecture (VHCO): data flows one direction and every dependency is injected, so each piece is testable in isolation. If you want to extend the binary — a new native feature, a new run mode — this is the map.

package layout
domain/        what the app is        (AppConfig, Message, RunMode)
features/      what it can do         (structs of capability functions)
usecases/      how it does things     (run modes, init-window)
infra/         external impls         (webview, sockets, http, wasm, native/*)
orchestrator/  the wiring             (builds & injects everything)
appio/         CLI + config loading

Features are structs of function values, not classes — the orchestrator constructs them by plugging in closures over infra. That makes any capability a value you can swap or fake. Full write-up: docs/architecture.md.

🔌 Backend protocol

The frontend ↔ backend wire format is newline-delimited JSON over a Unix socket (a named pipe on Windows). Each message carries an id to correlate replies; the final reply sets done: true. A backend pushes unsolicited events to the UI by using an id prefixed with server:.

messages on the socket
→  {"id":"a1","function":"add","data":{"value":5},"done":true}   // frontend asks
←  {"id":"a1","function":"add","data":{"value":6},"done":true}   // backend replies
←  {"id":"server:x","function":"tick","data":{"t":"12:30"}}      // backend pushes

On the frontend, the pushed event arrives via BACKEND.onEvent("tick", cb). A backend in any language only needs to read this format. Spec: docs/backend-protocol.md.

📦 What an app looks like on disk

myapp/
myapp/
├── window.yaml          # the app manifest
├── main.py              # backend (optional, any language)
└── static/
    ├── index.html       # your UI
    └── app.js

That folder is the app. Zip it, commit it, or hand it to a teammate — anyone with the window binary can run it. No install step, no lockfile.

✅ Testing & contributing

Because features are structs of functions, tests build them from fakes and assert on recorded calls — no mocking framework, no running webview. The repo ships unit tests per package plus end-to-end tests that build the binary and serve every demo over HTTP.

terminal
go build ./...      # compile everything
go test ./...       # unit + integration tests

window vs Electron, Tauri & Wails

All four put a web UI in a desktop window. They differ most in what language the backend is and how much tooling you take on.

When to pick window

  • Your backend logic already exists in Python, Go, Rust, or anything non-Node and you don't want to port it.
  • You want to ship or generate apps as plain files with no build step — internal tools, quick utilities, AI-authored apps.
  • You want camera/mic/speech/screen/input as one-liners out of the box.

When another tool fits better

  • Electron — you need the largest ecosystem, deep Chromium control, or guaranteed identical rendering on every OS.
  • Tauri / Wails — you're committed to Rust / Go and want that language compiled directly into one binary with its security model.
