The .window app language

Describe a whole native desktop app in one declarative file. window app.window transpiles it to a frontend and a window.yaml, then opens it. UI, events and state are generated into the page; any backend call is served in-process by Go handlers built into the binary. The transpiler under the hood is the Capy engine — you never see it.

how it works

One source, projected into a full app

A single statement contributes to several files at once — markup to the HTML, wiring to the JS, and config to the YAML — so nothing can drift. There is no separate backend process: BACKEND.call(...) dispatches to Go handlers inside the same binary.

60-second tour

A reactive counter, start to finish

Declare state, render it, wire two buttons to transitions. No backend, no build step — this is the whole app.

counter.window
app "Counter" 320 240
    heading "Counter"
    text count
    button "−" id dec
    button "+" id inc
    state count = 0
    on "#inc" click do up
    on "#dec" click do down
    action up   set count = state.count + 1
    action down set count = state.count - 1
end

Transpiles to a reactive store + event wiring (real output):

generated static/app.js (excerpt)
const state = { count: 0 };
function render(){
  document.querySelectorAll('[data-bind]').forEach(el => {
    const v = getp(state, el.dataset.bind);
    el.textContent = (v == null ? '' : v);
  });
}
function setState(patch){ Object.assign(state, patch); render(); }
const ACTIONS = {
  up:   () => setState({ count: state.count + 1 }),
  down: () => setState({ count: state.count - 1 }),
};
document.querySelector("#inc").addEventListener("click", ev => ACTIONS.up && ACTIONS.up(ev));
render();

in-process backend

Built-in Go handlers — no sidecar

Call these from any .window source with get / send. They run inside the window process (see infra/handlers.go) — no Python, no Node, no socket. Add your own there.

HandlerParamsReturns
time—{iso, unix, time, date}
uuid—{value}

sysinfo—{os, arch, cpus, hostname, go}
