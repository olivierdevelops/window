Reactive desktop apps in one .capyx file

A .capyx file is a whole reactive native app: dumb component views, handler units that own state and event logic, capability/provide boundaries for the outside world, and an optional orchestrator that owns shared data. window app.capyx compiles it to a window app with a fine-grained signals runtime inlined — mutate one field and only the bindings that read it re-run. No virtual DOM, no whole-tree re-render, no backend process required.

how it works

Three braces, four layers, one inlined runtime

The compiler (a pure text-transform in infra/) parses your components into a node tree and emits JavaScript that builds DOM through a tiny signals runtime. Each {{ expr }}, class:x={…} and attr={…} becomes its own effect; {#for}/{#if}/{#match} become dynamic regions whose control effect tracks only the loop or condition. There is no build step you run — window app.capyx does it all and opens the window.

60-second tour

A counter, start to finish

A component is a dumb view. A handler owns state and on-event methods that mutate only their own state. mount places the component with its handler. That's the whole program.

counter.capyx
app "Counter" 420x300

component counter
  <div class="card col">
    <h1>Counter</h1>
    <div class="row">
      <button class="btn" on:click={dec}>-</button>
      <span>{{ count }}</span>
      <button class="btn" on:click={inc}>+</button>
    </div>
    {#if count > 0}
      <p class="muted">{{ count }} and climbing.</p>
    {#elif count < 0}
      <p class="muted">below zero.</p>
    {#else}
      <p class="muted">back to zero.</p>
    {/if}
  </div>
end

handler counter
  state count = 0
  on inc()
    count = count + 1
  end
  on dec()
    count = count - 1
  end
end

mount counter
compiles to surgical bindings
// {{ count }} → its own effect
CAPYX.text(function(){ return H.count; });

// on:click={inc} → calls the handler method
CAPYX.on(btn,"click",function(){ return H.inc(); });

// {#if …} → control effect tracks only `count`
CAPYX.when(function(){
  return (H.count > 0) ? 0
       : (H.count < 0) ? 1 : 2;
}, [ /* one builder per branch */ ]);

// the handler is a reactive proxy:
//   count = count + 1   →   this.count = this.count + 1
// the assignment triggers exactly the effects
// that read `count` — the <span> and the {#if}.

the payoff

Fine-grained: mutate one field, touch one binding

The runtime is a small signals core: a reactive proxy track()s every field read inside an effect and trigger()s only those effects when the field changes. There is no diffing and no component re-render. In the dashboard demo, four progress bars each compile to their own style:width effect — bumping one bar re-runs that one effect and nothing else.

each bar is independent
{#for stat in stats}
  <div class="bar">
    <div class="fill"
      style:width={ stat.value + "%" }>
    </div>
  </div>
  <button on:click={bump}>+10</button>
{/for}
WHAT RE-RUNS ON bump(cpu) stat.value = stat.value + 10 ↓ trigger(cpu, "value") only the CPU bar's width effect re-runs. Memory / Disk / Network bars: untouched. The list itself: not re-rendered.

Lists are keyed: {#for item in items} reconciles by item.id, so adding, removing or reordering moves DOM nodes instead of rebuilding them — and each row keeps its own effects. {#if} and {#match} dispose the previous branch's effects when the branch key changes, so nothing leaks.

architecture

Four layers, closed-world (VHCO)

The format bakes in the VHCO discipline: each unit only touches what it owns, and the outside world is reachable through one explicit boundary. Views can't fetch; handlers can't reach the network directly; only a provide block does I/O; the orchestrator owns shared data and wiring.

component

View

Dumb HTML + three-brace bindings. No state, no I/O. Reads {{ field }} from its handler H.

handler

State + events

Owns state; on methods mutate only that state and call injected ports. Never touches the outside world.

provide

Capability

Implements a capability protocol. The only layer allowed to do real I/O. Injected into handlers as a port.

orchestrator

Shared data

Owns cross-screen data + boot. Auto-injectable as a capability by its own name; one reactive store for many handlers.

capability + provide (dependency injection)
capability Store
  fn seed()
end

provide Store as memStore
  fn seed()
    return [{id:1, text:"Buy milk"}]
  end
end

handler notes
  state items = []
  needs Store as store      # injected port
  on mount()
    items = store.seed()    # handler never does I/O itself
  end
end
orchestrator = one shared reactive store
orchestrator hub
  data
    count = 0
  end
  on bump()
    count = count + 1
  end
end

# one component, mounted twice with two handlers;
# both inject `hub`, so a click in either panel
# updates the shared count everywhere.
handler left for panel
  state bus = null
  needs hub as hub
  on mount()  bus = hub  end
  on inc()    hub.bump()  end
end

mount panel use right

Option-1 binding. Because a component is dumb and a handler is injected at mount, the same view can be mounted many times with independent handlers — mount tasklist use today and mount tasklist use work give two lists that share zero state. Swap in a stub handler to test a view in isolation.

format reference

Everything you can write

The three-brace rule

SyntaxMeaning
{{ expr }}Reactive text — escaped, re-runs when its reads change
attr={ expr }Reactive attribute (value/checked bind as properties)
class:name={ expr }Toggle a class when the expression is truthy
style:prop={ expr }Set one CSS property reactively
on:event={ handler }Event listener — a bare name calls H.name(item), or inline code
bind:value={ field }Two-way binding for inputs / <textarea> / <select>
{#for x in list} … {#else} … {/for}Keyed list with optional empty state
{#if c} {#elif c} {#else} {/if}Conditional region
{#match e}{#case v}…{#default}…{/match}Switch region

Top-level constructs

ConstructWhat it declares
app "Title" 460x520Window title + size
theme key="value"Design tokens → CSS variables ({{ theme.key }} in <style>)
component name … endA view; optional scoped <style> block inside
handler name [for comp] … endstate, needs … as … ports, on name() … end methods
capability Name … endA port protocol (signatures only)
provide Cap as name … endCapability implementation — fn name() … end
orchestrator name … enddata … end, on boot(), shared methods
mount comp [use handler] [as id]Place a component with a handler

Handler body mini-language

Method bodies are a tiny Python-flavoured language compiled to JavaScript: if/elif/else, for x in list, while, let, return, log, and bare expressions — each block closed with end. A bare reference to a state field is rewritten to flow through the reactive proxy, so count = count + 1 triggers exactly the bindings that read count. Arrow functions, await (auto-async) and standard globals (Math, JSON, setInterval…) all pass through.

which format

.capyx vs .window vs .htmlx

.capyx.window.htmlx
You writeReactive components + handlersHigh-level widgets + stateHTML markup by hand
ReactivityFine-grained signals (per-binding)Reactive store—
ArchitectureVHCO: view / handler / capability / orchestratorStore + actionsTags only
BackendNone needed — logic is in-fileIn-process Go handlers—
Best forInteractive apps, clean separation, testable viewsStateful apps with a Go backendStatic pages

All three are first-class window formats picked by file extension — no flags, no config.
