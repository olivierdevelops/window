Apps as <angle-bracket> HTML

A .htmlx file is a whole native desktop app written as real, matched-pair HTML — <tag>…</tag>. window app.htmlx parses it, validates every closing tag against its opener, wraps it in a normalized cross-platform document, and opens it. The parser is the Capy engine's sequence closers.

how it works

One generic rule parses every tag

A single Capy function opens on < + a captured tag name and closes on the sequence </ name >. Because the closer references the captured name, </div> closes only <div> — and a mismatch is a parse error. Before Capy runs, Go preprocessors expand <component> definitions and compile-time {#for} / {#if} / {#match} directives.

assets/htmlx.capy (the whole tag grammar)
function element
    arg literal "<"
    arg capture name ident
    arg capture attrs attribute*
    arg literal ">"
    block_close_seq "</" name ">"      # closer references the opener's name
    write `<${name}${attrs}>${body}</${name}>`
end

A landing page, start to finish

The root <app> sets the window title + size and wraps everything in a full document. Everything inside is plain, nested HTML. Text is quoted.

the payoff

Closing tags are checked against their openers

This is the one correctness win HTML's verbosity buys you, and you get it at transpile time. A closing tag that doesn't match the open tag it should is a hard error — not a silently mis-rendered page.

bad.htmlx
<app title="X" width="640" height="520">
  <div class="card"><p>"oops"</div>
</app>
TRANSPILE ERROR 2:28: no library function matches token "</" hint: did you mean "<"? <p> is never closed by </div>

The parser tracks every open tag and demands its own matching close. Unclosed tags at end-of-file and wrong closers are rejected the same way.

Built-in utility classes

The wrapper ships a cross-platform reset plus: .card, .row, .grid, .btn, .muted, .badge. Combine with inline style="…" for one-offs.

Looks the same on every OS & engine

The three webview engines — WKWebView (macOS), WebView2/Chromium (Windows) and WebKitGTK (Linux) — ship different UA defaults, system fonts, native form widgets and scrollbars. The .htmlx wrapper neutralizes those differences so a page renders near-identically everywhere:

  • Pinned metrics — explicit font-size, line-height, tab-size, box-sizing:border-box, and font-synthesis:none so missing font cuts never get faux-bolded differently per OS.
  • Form controls reset to zero — appearance:none on buttons and fields, a custom SVG <select> arrow, normalized number spinners and search decorations, a consistent ::placeholder color, and a single :focus-visible ring.
  • Native widgets tinted — accent-color gives checkboxes, radios and ranges the same color instead of the OS blue/green.
  • Styled scrollbars — identical thin scrollbars via ::-webkit-scrollbar (Chromium/WebKit) and scrollbar-width/-color, replacing macOS overlay vs Windows/Linux chunky bars.
  • Consistent type — a deep monospace stack for <pre>/<code>, antialiased smoothing, and a prefers-reduced-motion guard.

The one rule to remember

Text is quoted. One quoted string is one token, so multi-word text and even literal angle brackets live happily inside quotes. Bare unquoted prose between tags is not supported — if you want that, the .window language handles free prose with indentation instead.

metaprogramming

Define your own HTML tags

A .htmlx source can define new tags with a <component> block, then use them as if they were built in. It's all HTML: props="…" declares named attributes you read as {{ prop }} (escaped automatically, the same double-brace text binding as .capyx), and <slot> marks where nested children go. No JavaScript, no grammar to learn — <component> expands to a real tag at transpile time.

define it once — pure HTML
<component name="card" props="title">
  <section class="card">
    <h3>{{ title }}</h3>
    <slot></slot>
  </section>
</component>
use it like built-in HTML
<card title="Welcome">
  <badge>"New"</badge>
  <p>"Children fill the slot."</p>
</card>
EXPANDS TO <section class="card"> <h3>Welcome</h3> <span class="badge">New</span> <p>Children fill the slot.</p> </section>

Props render escaped, so components are XSS-safe by default; add the bare void attribute for a self-closing tag like <component name="avatar" props="name" void> → <avatar name="Ada" />. The components.htmlx demo defines <card>, <stat>, <badge> and <avatar/> and builds a dashboard from them.

compile-time

Control flow — loop, branch, match

The same three-brace directives as .capyx run at transpile time: {#for}, {#if}/{#elif}/{#else}, and {#match}/{#case}/{#default}, each closed with {/…}. There's no runtime data model — each construct is evaluated while the file compiles, so the shipped page is plain static HTML. They compose: an {#if} inside a {#for} sees the loop variable as {{ name }}.

loop, branch & match
{#for role in admin, editor, guest}
  <section class="card">
    <h3>"{{ role }}"</h3>
    {#match role}
      {#case admin}
        <p>"Full control."</p>
      {#case editor}
        <p>"Create & edit."</p>
      {#default}
        <p>"View only."</p>
    {/match}
  </section>
{/for}
EXPANDS TO <section class="card"><h3>admin</h3> <p>Full control.</p></section> <section class="card"><h3>editor</h3> <p>Create & edit.</p></section> <section class="card"><h3>guest</h3> <p>View only.</p></section>

{#for x in a, b, c} repeats its body once per comma-separated item; {#if x == y} picks a branch (use {#if x in a, b} to test list membership, plus optional {#elif} / {#else}); {#match v} runs the first matching {#case …} or the {#default}. Each directive sits on its own line. The control.htmlx demo builds a nav, a banner and a role grid this way.
